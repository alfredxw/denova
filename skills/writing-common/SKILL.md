# 写作公共规范

本 Skill 定义所有写作流程（novel-lite / novel-standard / novel-heavy / continue / rewrite）共享的基础规则。具体流程 Skill 在此基础上定义各自的流程差异。

## 写作范围判断

- 从用户的实际指令判断写作范围，例如"续写一段""写一个场景""写一章""写三章""写一个剧情 arc"或用户自定义目标。
- 除非用户明确说"写下一章"，否则不要假设任务一定是下一章。
- 没有 `writing_scope` 字段；不要等待或编造额外字段。
- 用户消息是判断范围、目标、约束和输出形态的唯一来源。
- 当用户要求一次写 N 章或多段 arc 时，先制定整体计划和分章计划。

## 上下文读取

写作前使用 `read_file` 读取必要上下文：

- `CREATOR.md`（如存在）
- `setting/outline.md`
- `setting/progress.md`
- `setting/character-states.md`
- 相关章节组细纲（`setting/chapter-groups/`）
- 最近章节正文（至少 2 章）
- 涉及资料库条目时先用 `list_lore_items` 判断相关性，再用 `read_lore_items` 读取完整资料

只读取与本轮写作范围相关的上下文，不要无界读取全部文件。

## 工具使用规范

- 创建新章节、写入整章初稿、覆盖整章修订稿时使用 `write_file`。
- 局部修改已有章节或状态文件时使用 `edit_file`，并确保 `old_string` 来自最近一次 `read_file` 的实际内容且不包含行号前缀。
- 更新 `setting/progress.md` 和 `setting/character-states.md` 时，优先用 `edit_file` 更新对应条目；只有文件不存在、结构严重不匹配或需要全量重排时才用 `write_file`。
- 需要精确统计字数时（如字数门控、篇幅检查），使用 `count_words` 工具而非模型自行估算。传入 `file_path` 统计文件，或传入 `text` 统计内联文本。以返回结果中的 `chinese_chars` 作为"中文字数"。不要依赖 token 计数或模型对字数的猜测。

## 写入错误检查

每次调用 `write_file` 或 `edit_file` 后都要检查工具结果：

- 若结果包含 `[tool error]`、参数 JSON 错误、`string not found`、路径错误或截断提示，不得宣称已完成。
- 应重新读取目标文件、修正参数后重试，或明确告诉用户未写入成功。

## 读回验证

对用户可见的最终文件改动，完成后使用 `read_file` 读回目标文件的关键片段，确认内容已经落盘。如果写入了状态文件，也读回对应关键片段。无法验证时，在最终回复中说明未完成验证。

## 章节正文格式

- 章节正文使用纯文本自然段，段落间空行分隔。
- 禁止使用 Markdown 标题、列表、引用、加粗等标记。
- 章节开头直接开始叙事，不要输出章节标题。

## 状态文件边界

- `setting/outline.md`：只记录长期故事结构、主线方向、卷章安排和章节目标。不记录已写进度、最近章节复盘或角色当前状态。除非作者明确要求调整大纲，不更新此文件。
- `setting/progress.md`：只记录当前写作进度、已完成章节摘要和短期衔接提示。不承担长期大纲职责。
- `setting/character-states.md`：只记录角色当前状态（最近出场、当前位置、身体/心理状态、当前目标、持有物、能力变化、关系变化、待回收伏笔）。不写未来规划。
- 资料库（lore）：只记录角色身份、人设、长期关系、能力体系、世界规则、地点、势力、物品等稳定设定。不写章节规划、未来剧情或每章状态抖动。
- 未确认成章的初稿不写入 `progress.md`、`character-states.md` 或资料库。
- 只有作者明确确认成章或明确要求同步状态后，才更新状态文件。
- 只有长期设定发生明确变化时，才使用 `write_lore_items` 同步资料库。

## 最终输出

- 返回最终正文或用户要求的写作产物。
- 只有任务产生了可持久化进展，或用户要求说明时，才附带简短状态更新摘要。
- 除非用户明确要求检查流程，否则隐藏内部角色对话和审稿过程。
- 如果关键约束无法满足，先简短说明阻断问题或请求用户确认。
