# novel-under-plan

严格按计划逐分场写作的协作流程。先加载 `writing-common` 获取公共规范（工具使用、错误检查、读回验证、章节正文格式、状态文件边界），再按本 Skill 执行。

本 Skill 采用多角色审稿管线（context-planner / writer / reviewer / fixer / memory-patcher），将写作粒度收窄到**单个分场（scene）**，并在每个阶段增加交互确认点。定位介于 Plan Mode（只规划不执行）和普通写作模式（直接执行）之间：**会写作的 Plan / 会提问的 Agent**。

本 Skill 是自包含的，不依赖其他 novel-* 流程 Skill。选中本 Skill 后，不要加载 `novel-heavy`、`novel-standard` 或 `novel-lite`。

## 核心原则

1. **计划即法律**：粗纲 / 章节组细纲 / 用户确认的分场计划是唯一写作依据。不得偏离计划自行发挥、加戏或跳过 beats。
2. **一次一场**：每轮只精细写作一个分场。不批量生成多个分场或整章。
3. **字数门控**：分场字数未达到计划要求前不得退出该分场。未达标时继续扩写、补充细节或深化冲突，而非草草收尾。
4. **高交互性**：在关键决策点主动向用户提问确认，而非默默假设。像协作伙伴而非自动执行器。

## 流程概览

```
定位分场 -> 分场计划确认 -> 写作 -> 审稿 -> 修订 -> 字数检查 -> 用户确认 -> 状态更新 -> 下一场提示
```

## 阶段详解

### 阶段 1：定位当前分场

1. 读取 `setting/outline.md`、`setting/progress.md`、`setting/character-states.md`。
2. 如存在 `setting/chapter-groups/` 下当前章节组细纲，读取并定位当前应写的分场。
3. 如用户明确指定了分场（如"写第三章第二场"），直接使用用户指定的分场。
4. 读取前面至少 2 章定稿正文（或当前章已完成分场），确保衔接。
5. 涉及资料库条目时先用 `list_lore_items` 判断，再用 `read_lore_items` 读取。

**交互点**：向用户确认当前要写的分场编号、所属章节和分场目标。如果计划中分场划分不清晰，先提出分场拆分建议供用户确认。

### 阶段 2：分场计划确认

基于粗纲/细纲中该分场的安排，生成分场写作计划：

```md
# 分场计划

## 定位
- 所属章节：第X章
- 分场序号：第Y场
- 前场落点：（上一场结束时的状态）

## 本场目标
（这一场要完成的剧情功能，来自粗纲/细纲）

## Required Beats
（必须发生的关键事件，逐条列出，来自计划）

## 字数目标
（本场目标字数，如 2000-3000 字）

## 角色状态
（出场角色当前状态、动机、已知信息）

## Canon 约束
（不能违背的世界观、时间线、伏笔等）

## 风格约束
（叙事人称、文风、节奏）

## 风险
（本场最容易写崩的地方）
```

**交互点**：将分场计划展示给用户确认。用户可以调整 beats、字数目标或风格要求。只有用户确认后才进入写作。

如果计划中存在模糊或矛盾之处（如粗纲描述与角色当前状态冲突），必须先向用户提问澄清，不要自行假设。

### 阶段 3：写作

按确认后的分场计划写作正文。

- 严格覆盖计划中的每一个 Required Beat，不遗漏、不自行新增。
- 使用 `write_file` 写入 `chapters/` 下对应章节文件（新章节）或 `edit_file` 追加到已有章节末尾。
- 正文格式遵循 `writing-common` 的章节正文格式规范。
- 写作过程中如发现计划与 canon 存在不可调和的冲突，暂停写作并向用户报告，等待指示。

如果 `writer` subagent 可用，通过 `task` 工具委派，在 description 中写清分场计划全文、字数目标、允许/禁止写入和期望输出格式。

### 阶段 4：审稿

如果 `reviewer` subagent 可用，通过 `task` 工具委派审稿。审稿重点：

- **计划符合度**（最高优先级）：每个 Required Beat 是否已覆盖？是否有计划外的加戏或偏离？
- 连续性：与前场和前文的衔接是否自然？
- 角色声线：角色行为和对白是否符合当前状态？
- 节奏和文风：是否符合风格约束？

reviewer 必须返回结构化问题，每项包含：

- `severity`: `blocker` / `major` / `minor`
- `dimension`: `plan_compliance` / `continuity` / `character_voice` / `pacing` / `prose` / `dialogue` / `plot_logic` / `style` / `user_requirement`
- `problem`
- `fix_instruction`
- `keep`

其中 `plan_compliance` 是本 Skill 特有的最高优先级维度：检查每个 Required Beat 是否已覆盖、是否有计划外加戏或偏离。

### 阶段 5：修订

根据审稿结论修订，只修真正需要修的问题。如果存在 `plan_compliance` 维度的 blocker（如遗漏 beat 或偏离计划），必须优先修复。

### 阶段 6：字数检查

使用 `count_words` 工具（而非模型自行估算）统计本场正文的精确中文字数：

- 如果正文已写入文件：调用 `count_words` 并传入 `file_path`（章节文件绝对路径）。
- 如果正文尚未写入文件：调用 `count_words` 并传入 `text`（本场正文全文）。
- 以返回结果中的 `chinese_chars` 字段作为"中文字数"判定依据。不要依赖模型对字数的估算或 token 计数。

判定规则：

- **达标**（`chinese_chars` ≥ 计划字数下限）：进入用户确认阶段。
- **未达标**：不得退出，进入扩写交互流程。
- 每次扩写或修订后，必须重新调用 `count_words` 获取最新字数，不得沿用旧计数。

**未达标扩写交互**：

字数不足时，不要自行决定扩写方向。向用户报告当前字数与目标差距，并询问希望增加哪类描写。提供以下选项供用户选择（可多选）：

| 扩写方向 | 说明 |
|----------|------|
| 外貌描写 | 角色外貌、神态、穿着、体态等视觉细节 |
| 动作描写 | 角色肢体动作、微表情、行为细节 |
| 心理描写 | 角色内心独白、情绪波动、思绪流转 |
| 回忆/闪回 | 与当前情境相关的角色过往记忆片段 |
| 环境描写 | 场景氛围、天气、光线、声音、气味等感官细节 |
| 对话扩展 | 深化角色对白、增加潜台词或对话中的博弈 |
| 冲突深化 | 加剧当前矛盾、增加阻力或两难抉择 |
| 过渡铺垫 | 在计划 beats 范围内增加衔接和节奏过渡 |

默认回退值：如果用户未选择或表示"你来决定"，参考当前叙事方案（Teller / Narrative Style）的风格倾向自动选择最匹配的 1-2 个方向。例如：
- 经典叙事 / 传统文学风格 → 优先「心理描写」「环境描写」
- 轻小说 / 网文风格 → 优先「对话扩展」「冲突深化」
- 影视化 / 画面感风格 → 优先「动作描写」「环境描写」

如无法确定当前叙事方案，默认选择「心理描写」+「环境描写」。

约束：
- 扩写必须在计划 beats 范围内进行，不得新增计划外情节。
- 不得通过注水、重复或无关描写凑字数。
- 如果经过 2 轮扩写仍未达标，向用户报告实际字数和差距，询问是调整字数目标还是继续扩写。

### 阶段 7：用户确认

将完成的分场正文展示给用户，附带：

- 本场覆盖的 Required Beats 清单（逐条标注 ✅）
- 实际字数 vs 目标字数
- 审稿发现及修复摘要（如有）

等待用户确认"通过"或提出修改意见。用户提出修改意见时，回到阶段 5 修订后重新走字数检查和确认。

### 阶段 8：状态更新

用户确认后，按 `writing-common` 的状态文件边界规则：

- 更新 `setting/progress.md`（当前进度 + 本场摘要）。
- 更新 `setting/character-states.md`（角色状态变化）。
- 只有长期设定发生明确变化时，才使用 `write_lore_items` 同步资料库。
- 不更新 `setting/outline.md`。

如果 `memory-patcher` subagent 可用，通过 `task` 工具委派生成状态 patch。

### 阶段 9：下一场提示

状态更新完成后，向用户提示：

- 下一场的编号、所属章节和计划目标（如计划中已有）。
- 询问用户是否继续写下一场，或暂停。

不自动开始下一场写作。

## 交互协议

本 Skill 的交互性高于其他写作模式。在以下场景**必须**向用户提问或确认：

| 场景 | 交互方式 |
|------|----------|
| 分场划分不清晰 | 提出拆分建议，等用户选择 |
| 计划与 canon 冲突 | 报告冲突，等用户决策 |
| 分场计划确认 | 展示计划，等用户确认或调整 |
| 字数 2 轮扩写仍未达标 | 报告差距，询问调整目标或继续 |
| 分场完成 | 展示正文 + beats 清单，等用户确认 |
| 下一场开始 | 提示下一场信息，等用户确认继续 |

提问时保持简洁，一次只问关键问题，不要堆砌确认项。用户明确表示"不用确认，直接写"时，可跳过非关键确认点（分场计划确认和最终确认仍保留）。

## 与其他模式的关系

- 本 Skill 与 `novel-heavy`、`novel-standard`、`novel-lite` 互斥：选中本 Skill 后不加载其他 novel-* 流程 Skill，公共规范统一来自 `writing-common`。
- 比 `novel-heavy` 粒度更细（分场 vs 章/arc），交互确认点更多。
- 比 Plan Mode 多执行能力（会写作、会写入文件）。
- 如果用户只要求"写一章"而非逐场推进，建议用户切换到 `novel-heavy` 或 `novel-standard`。
- 本 Skill 适用于：用户有明确粗纲/细纲，希望逐场精细推进、深度参与创作决策的场景。
